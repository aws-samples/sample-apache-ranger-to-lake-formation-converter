# Production / Demo Deployment

Deploy the Ranger-to-Lake Formation Conversion Server against real AWS Lake Formation APIs using Maven profiles and Docker Compose.

## How This Differs from `integration-test/docker/`

| Aspect | `integration-test/docker/` | `deploy/` |
|--------|---------------------------|-----------|
| Purpose | Automated integration tests | Production / demo deployment |
| `DRY_RUN_ENABLED` | `true` — uses `DryRunLakeFormationClient` | `false` — calls real Lake Formation APIs |
| AWS credentials | Not required | Required (passed via Maven `-D` properties) |
| Config file | `server-config-it.yaml` | `server-config-deploy.yaml` (generated by Maven) |
| Dry-run output | Writes to `dry-run-output/` volume | Not applicable |

Both stacks share the same root `Dockerfile` and Ranger Admin infrastructure (PostgreSQL, Solr, Ranger Admin). Only the Conversion Server runtime configuration differs.

## Prerequisites

- Docker with Docker Compose v2
- Maven 3.6+
- An AWS account with Lake Formation and Glue permissions
- IAM credentials (access key / secret key) or an IAM role with appropriate permissions
- At least one Ranger-to-IAM principal mapping ready

## Architecture

Two Maven profiles manage independent Docker Compose stacks:

- **`deploy-server`** — builds and starts the conversion-server container only (`docker-compose-server.yml`)
- **`deploy-ranger-admin`** — starts the Ranger Admin stack: ranger-db, ranger-solr, ranger-admin (`docker-compose-ranger-admin.yml`)

Both stacks share a `rangernw` Docker network, so the conversion-server can reach Ranger Admin when both are running.

> **Key change from the previous workflow:** You no longer edit `server-config-deploy.yaml` or `.env` by hand. Maven generates both files from `-D` command-line properties before starting Docker Compose.

## Quick Start

### Deploy the Conversion Server (with an existing Ranger Admin)

All commands are run from the **project root** (where `pom.xml` lives).

**Start the server:**
```bash
mvn generate-resources exec:exec@ensure-network exec:exec@docker-compose-up \
  -Pdeploy-server \
  -Daws.account.id=123456789012 \
  -Daws.role.arn=arn:aws:iam::123456789012:role/LakeFormationRole \
  -Daws.access.key.id=EXAMPLE_ACCESS_KEY_ID \
  -Daws.secret.access.key=EXAMPLE_SECRET_ACCESS_KEY
```

**Stop the server:**
```bash
mvn exec:exec@docker-compose-down -Pdeploy-server
```

### Deploy a Local Ranger Admin Stack

If you don't have a running Ranger Admin instance, start one locally first:

**Start Ranger Admin:**
```bash
mvn exec:exec@ensure-network exec:exec@docker-compose-up-ranger -Pdeploy-ranger-admin
```

**Stop Ranger Admin:**
```bash
mvn exec:exec@docker-compose-down -Pdeploy-ranger-admin
```

### Combined Usage (Local Ranger Admin + Conversion Server)

When you need both stacks running together:

```bash
# 1. Start Ranger Admin stack first
mvn exec:exec@ensure-network exec:exec@docker-compose-up-ranger -Pdeploy-ranger-admin

# 2. Wait for Ranger Admin to become healthy (check http://localhost:6080)

# 3. Start the conversion server (pointing to local Ranger Admin)
mvn generate-resources exec:exec@ensure-network exec:exec@docker-compose-up \
  -Pdeploy-server \
  -Daws.account.id=123456789012 \
  -Daws.role.arn=arn:aws:iam::123456789012:role/LakeFormationRole \
  -Daws.access.key.id=EXAMPLE_ACCESS_KEY_ID \
  -Daws.secret.access.key=EXAMPLE_SECRET_ACCESS_KEY

# 4. Stop both stacks when done
mvn exec:exec@docker-compose-down -Pdeploy-server
mvn exec:exec@docker-compose-down -Pdeploy-ranger-admin
```

The default `ranger.admin.url` is `http://ranger-admin:6080`, which resolves via Docker DNS on the shared `rangernw` network. No override is needed when both profiles are running together.

## Maven Properties Reference

### Required Properties (deploy-server only)

| Property | Description | Example |
|----------|-------------|---------|
| `aws.account.id` | 12-digit AWS account ID. Maps to `awsConfig.catalogId` in config YAML. | `123456789012` |
| `aws.role.arn` | IAM role ARN for Lake Formation. Maps to `awsConfig.roleArn`. | `arn:aws:iam::123456789012:role/LFRole` |
| `aws.access.key.id` | AWS access key ID. Written to `.env` as `AWS_ACCESS_KEY_ID`. | `EXAMPLE_ACCESS_KEY_ID` |
| `aws.secret.access.key` | AWS secret access key. Written to `.env` as `AWS_SECRET_ACCESS_KEY`. | `EXAMPLE_SECRET_ACCESS_KEY/...` |

If any required property is missing, the build fails immediately with a descriptive error (enforced by `maven-enforcer-plugin`).

### Optional Properties

| Property | Default | Used By | Description |
|----------|---------|---------|-------------|
| `aws.region` | `us-east-1` | Config YAML + `.env` | AWS region for Lake Formation and Glue API calls |
| `ranger.admin.url` | `http://ranger-admin:6080` | Config YAML | Ranger Admin endpoint URL |
| `ranger.admin.password` | `rangerR0cks!` | Config YAML + `.env` | Ranger Admin authentication password |
| `ranger.version` | `2.4.0` | Compose `RANGER_VERSION` | Apache Ranger Docker image version |
| `principal.user.mappings` | _(empty)_ | Config YAML | Comma-separated `name=arn` user mappings |
| `principal.group.mappings` | _(empty)_ | Config YAML | Comma-separated `name=arn` group mappings |
| `principal.role.mappings` | _(empty)_ | Config YAML | Comma-separated `name=arn` role mappings |
| `sync.interval.ms` | `30000` | Config YAML | Policy refresh interval in milliseconds |
| `lf.max.retries` | `5` | Config YAML | Max Lake Formation API retries |
| `lf.retry.backoff.ms` | `2000` | Config YAML | Backoff between retries in milliseconds |
| `server.log.level` | `INFO` | Config YAML | Server log level (DEBUG, INFO, WARN, ERROR) |
| `server.shutdown.timeout` | `30` | Config YAML | Graceful shutdown timeout in seconds |

### Principal Mapping Format

Principal mappings are comma-separated `name=arn` pairs:

```bash
-Dprincipal.user.mappings=alice=arn:aws:iam::123:user/Alice,bob=arn:aws:iam::123:user/Bob
-Dprincipal.group.mappings=analysts=arn:aws:iam::123:role/AnalystRole
-Dprincipal.role.mappings=admin_role=arn:aws:iam::123:role/AdminRole
```

These are parsed into YAML map entries in the generated `server-config-deploy.yaml`.

## How It Works

1. **Validate** — `maven-enforcer-plugin` checks that all required `-D` properties are present.
2. **Parse mappings** — `gmavenplus-plugin` converts comma-separated principal mappings into YAML map syntax.
3. **Generate config** — `maven-resources-plugin` filters template files from `src/main/deploy/` into `deploy/`, replacing `${...}` placeholders with property values.
4. **Docker network** — `exec-maven-plugin` runs `docker network create rangernw` (idempotent — succeeds even if the network already exists).
5. **Docker Compose up** — `exec-maven-plugin` runs `docker compose up` with the appropriate compose file.

The generated files (`deploy/server-config-deploy.yaml` and `deploy/.env`) are git-ignored. The source-of-truth templates live in `src/main/deploy/`.

## Verifying the Deployment

### Health Checks

Docker Compose runs health checks on all services automatically. Check status from the `deploy/` directory:

```bash
docker compose -f docker-compose-server.yml ps
docker compose -f docker-compose-ranger-admin.yml ps
```

Expected healthy services:

- **deploy-server profile:** `conversion-server`
- **deploy-ranger-admin profile:** `ranger-db`, `ranger-solr`, `ranger-admin`

### Ranger Admin UI

Open `http://localhost:6080` in a browser (credentials: `admin` / your configured password) to verify Ranger Admin is up and policies are visible.

## Logs and Troubleshooting

### Viewing Logs

Stream logs from the conversion server:
```bash
docker compose -f deploy/docker-compose-server.yml logs -f conversion-server
```

Stream logs from the Ranger Admin stack:
```bash
docker compose -f deploy/docker-compose-ranger-admin.yml logs -f
```

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Build fails with "property is required" | Missing a required `-D` property | Add the missing `-Daws.account.id=...` (or whichever property) to the command |
| `conversion-server` exits immediately | Invalid generated YAML | Check `deploy/server-config-deploy.yaml` for unresolved `${...}` tokens. Ensure all required properties were passed. |
| AWS authentication errors in logs | Invalid or expired credentials | Verify `aws.access.key.id` and `aws.secret.access.key` values are correct and active. |
| `AccessDeniedException` from Lake Formation | IAM role lacks permissions | Ensure the role in `aws.role.arn` has `lakeformation:*` and `glue:Get*` permissions. |
| `ranger-admin` never becomes healthy | Port conflict or resource constraints | Check if port 6080 is already in use. Increase Docker memory allocation. |
| `conversion-server` can't reach `ranger-admin` | Not on the same Docker network | Both stacks must use the `rangernw` network. Run `docker network ls` to verify it exists. |
| Policies not syncing | No principal mappings configured | Add at least one `-Dprincipal.user.mappings=name=arn` entry. |
| `ConcurrentModificationException` in logs | Concurrent Lake Formation updates | Retried automatically (up to `lf.max.retries` times). Usually transient. |
